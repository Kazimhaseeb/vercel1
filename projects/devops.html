PLATFORM 01 / AMAZON WEB SERVICES

Amazon Web Services (AWS)

EC2 instances host long-running Python bots and APIs, S3 stores static assets and data exports, RDS provides managed PostgreSQL databases, and Lambda handles lightweight scheduled tasks — all within a single AWS account.

EC2 S3 RDS PostgreSQL Lambda IAM Roles CloudWatch

AWS is used for projects requiring persistent, scalable infrastructure. EC2 t3.micro / t3.small instances run trading bots and FastAPI services in Docker containers with auto-restart policies. S3 buckets store scraped datasets, Excel exports, OCR outputs, and static frontend assets served via CloudFront. RDS (PostgreSQL) provides a managed, auto-backed-up database layer for Django and FastAPI backends, removing the need to self-manage Postgres. Lambda functions handle lightweight scheduled jobs — price checks, digest emails, report generators — eliminating the need for always-on servers for simple cron tasks. All AWS resources are managed under IAM roles with least-privilege policies, and CloudWatch monitors logs and fires alarms on error spikes.

How AWS Is Used Across Projects

1
EC2 for Trading Bots & APIs
EC2 t3.micro instances run Dockerised trading bots and FastAPI services 24/7. Elastic IPs give a fixed public address. Security groups restrict inbound traffic to only required ports. Instances use IAM instance profiles instead of hardcoded credentials.
2
S3 for Data & Static Assets
Scraped job listings, OCR outputs, Excel trade journals, and CSV datasets are uploaded to private S3 buckets using boto3. Static portfolio assets and frontend builds are served from public S3 buckets via CloudFront CDN for fast global delivery.
3
RDS PostgreSQL for Backends
Django and FastAPI projects use RDS PostgreSQL for a fully managed database — automatic daily backups, point-in-time recovery, and Multi-AZ option for high availability. Connection pooling via pgBouncer reduces idle connection overhead.
4
Lambda for Scheduled Tasks
Lightweight cron jobs — daily digest emails, price alert checks, news scrape triggers — are deployed as Lambda functions triggered by EventBridge (CloudWatch Events) rules. No server needed; pay only for actual execution time.
5
CloudWatch Monitoring & Alarms
CloudWatch collects EC2 CPU, memory (via CloudWatch Agent), and Lambda error metrics. Alarms trigger SNS notifications and Telegram alerts when error rates exceed thresholds or instances go unhealthy.
Security Best Practices

All AWS credentials are managed via IAM roles — no access keys in code or environment files on instances. S3 bucket policies enforce least-privilege access. Security groups allow only ports 22 (SSH, key-based only), 80, and 443. All secrets (API keys, DB passwords) are stored in AWS Secrets Manager and fetched at runtime.

PLATFORM 02 / DIGITALOCEAN

DigitalOcean

DigitalOcean Droplets provide affordable Linux VPS instances for running bots and web services, the App Platform offers a simpler PaaS deploy option, and Managed Databases handle PostgreSQL without self-management overhead.

Droplets App Platform Managed DB Spaces (S3-compatible) Nginx VPS Monitoring

DigitalOcean is the platform of choice for straightforward Linux VPS deployments where direct server access is needed. Droplets (Ubuntu 22.04 LTS, $6–$12/month) run Docker containers for trading bots and Python services with full SSH control. The App Platform is used for simpler web apps — it auto-detects the Dockerfile or Procfile, handles builds, and deploys on every git push to main, comparable to Heroku but with better pricing. Managed PostgreSQL on DigitalOcean eliminates DBA work — automated backups, failover, and connection pooling are all built in. Spaces provides S3-compatible object storage for datasets and static files. DigitalOcean's built-in monitoring graphs CPU, RAM, disk I/O, and bandwidth for every Droplet with configurable alert policies.

PLATFORM 03 / RAILWAY · HEROKU · RENDER

Railway, Heroku & Render

PaaS platforms used for fast, zero-config deployments of web apps and APIs — ideal for client projects and prototypes where speed of deployment matters more than fine-grained server control.

Railway Heroku Render Zero-Config Deploy Auto SSL PostgreSQL Add-ons

Railway is the primary PaaS for Django and FastAPI web projects. It connects directly to the GitHub repo, detects the Dockerfile automatically, builds and deploys on every push to main, and provides a managed PostgreSQL database with a single click. Deployments are zero-downtime rolling updates. Heroku is used for simpler Flask and Django apps, especially when the Heroku Postgres add-on or free-tier Dyno simplifies early-stage prototyping. Render is used for FastAPI microservices and static site hosting — its free tier keeps small automation APIs live at no cost, and its paid tier offers background workers for Celery queues. All three platforms handle SSL certificates automatically on custom domains.

PLATFORM 04 / DOCKER · DOCKER COMPOSE

Docker & Docker Compose

Every project is containerised with Docker for environment consistency across development, staging, and production. Docker Compose orchestrates multi-container setups — web server, database, Redis, and background workers in a single command.

Dockerfile Multi-Stage Builds Docker Compose restart: always Docker Hub Volume Mounts

Docker is the foundation of every deployment. Every Python project has a Dockerfile using a pinned base image (e.g., python:3.11-slim) with explicit dependency installation, a non-root user, and a health check endpoint. Multi-stage builds keep production images lean by separating the build environment from the runtime image. Docker Compose defines the full stack for local development and VPS deployments: the app container, PostgreSQL, Redis (for Celery), and Nginx in a single docker-compose.yml. All containers use restart: always so they automatically recover from crashes or VPS reboots. Images are pushed to Docker Hub and tagged with git commit SHAs for version traceability. Volume mounts persist databases and trade history files across container restarts.

Docker Best Practices Used

Pinned base images — python:3.11-slim with exact version pins prevents "works on my machine" issues
Non-root user — all containers run as a dedicated non-root user for security isolation
Layer caching — COPY requirements.txt before code copy maximises Docker layer cache hits
Health checks — HEALTHCHECK instruction ensures container restarts if the app becomes unresponsive
Volume mounts — databases and trade history persist across container updates via named Docker volumes
Environment files — secrets injected via .env files, never baked into the image

PLATFORM 05 / GITHUB ACTIONS

GitHub Actions — CI/CD Pipeline

A fully automated CI/CD pipeline runs on every push to main — linting with flake8, testing with pytest, building and pushing a Docker image, then triggering a zero-downtime deploy on the target platform, all within ~3 minutes.

GitHub Actions pytest flake8 Docker Build Zero-Downtime Telegram Alerts

Every project repository has a GitHub Actions workflow (.github/workflows/deploy.yml) that fires on every push to the main branch. The pipeline runs in approximately 3 minutes end-to-end: flake8 checks for style violations, pytest runs the full test suite in a matrix across Python 3.10 and 3.11, the Docker image is built with layer caching and pushed to Docker Hub with both a latest and a commit-SHA tag, and finally the target platform (Railway, EC2, or DigitalOcean) is triggered to pull and deploy the new image. A Telegram notification reports deploy success or failure with a direct link to the failed step.

Pipeline Steps

1
Push to main → Workflow Triggered
Developer pushes to main branch (or a PR is merged). GitHub Actions workflow triggers within seconds, checking out the code and setting up the Python environment.
2
Lint & Test (Parallel Matrix)
flake8 checks for PEP8 style violations. pytest runs the full test suite (unit + integration) in a matrix across Python 3.10 and 3.11. Any failure immediately blocks the pipeline and sends a Telegram alert.
3
Docker Build & Push to Hub
Multi-stage Docker image is built with layer caching (via actions/cache for Docker layers) keeping build times under 90 seconds. The image is pushed to Docker Hub with latest and a commit-SHA tag for rollback ability.
4
Deploy to Target Platform
For Railway: the Railway CLI triggers a redeploy. For EC2/DigitalOcean: SSH action pulls the new image and runs docker-compose up -d --no-deps app for a zero-downtime rolling update. Health check verifies the new container is healthy before marking success.
5
Telegram Notification
On success: [SUCCESS] "Deployed [service] — commit abc1234" is sent to the team Telegram channel. On failure: [FAILURE] "Pipeline failed at [step] — [link to logs]" fires immediately with the failing step identified.

PLATFORM 06 / NGINX · CERTBOT · LET'S ENCRYPT

Nginx, SSL & Reverse Proxy

Nginx acts as the production web server for all VPS-hosted services — handling reverse proxying, SSL termination via Certbot/Let's Encrypt, gzip compression, rate limiting, and static file serving for all projects on a single server.

Nginx Certbot Let's Encrypt SSL Reverse Proxy Rate Limiting Gzip Compression

On every VPS deployment (EC2 or DigitalOcean Droplet), Nginx runs as a Docker container (or system service) and acts as the single entry point for all web traffic on ports 80 and 443. It reverse-proxies requests to the correct upstream Docker container based on the server_name (domain). Certbot with the DigitalOcean / Route53 DNS plugin automatically provisions and renews Let's Encrypt SSL certificates for all domains, with auto-renewal via a cron job. Gzip compression reduces response sizes by 60–70% for JSON APIs. Rate limiting is configured per-IP to protect APIs from abuse. Static files (collected Django assets, frontend bundles) are served directly from Nginx without hitting the Python process, significantly reducing server load.

Nginx Config Features

  • Reverse proxy to Docker upstream containers
  • HTTPS redirect from port 80 to 443
  • SSL termination with TLS 1.2 / 1.3 only
  • HSTS headers for browser security
  • Gzip on JSON, HTML, CSS, JS responses
  • Static file serving with long cache headers

Security Headers Set

  • Strict-Transport-Security (HSTS)
  • X-Frame-Options: DENY
  • X-Content-Type-Options: nosniff
  • X-XSS-Protection: 1; mode=block
  • Content-Security-Policy header
  • Referrer-Policy: no-referrer
Auto SSL Renewal

Certbot is configured with a cron job that runs certbot renew --quiet twice daily. Let's Encrypt certificates are valid for 90 days and auto-renewed at 30 days remaining. Nginx is reloaded (not restarted) on renewal to serve the new cert without downtime: certbot renew --deploy-hook "nginx -s reload".

Deployed to Production

Every project in this portfolio ships with a production deployment — not just a local demo. AWS, DigitalOcean, Railway, Docker, GitHub Actions, and Nginx form the infrastructure backbone that keeps all projects live, secure, and automatically updated.

Production Philosophy Every bot and API is containerized with explicit dependency pinning, non-root Docker users, and health check endpoints. Secrets are managed via environment variables and AWS Secrets Manager — never hardcoded. Logging is structured as JSON for easy aggregation. Restart policies ensure 24/7 uptime even after crashes or reboots.
